# IncidentCopilot 产品需求文档

## 1. 文档状态

| 字段 | 值 |
| --- | --- |
| 状态 | Current 产品边界（简历最终版） |
| 更新日期 | 2026-07-20 |
| 目标版本 | 作品集 MVP |
| 主要用户 | AI 应用开发岗位面试官、候选人、学习者 |

本文使用三个固定状态标签：**Current** 表示默认或明确可运行链路；**Experimental**
表示已有隔离实现但未进入默认链路或缺少 live 集成证据；**Target** 表示未来方向，不能
作为当前能力陈述。

## 2. 问题与机会

微服务故障调查需要在日志、指标、Trace、服务拓扑、发布变更、Runbook 和历史事故之间建立时间与因果关联。常见的“把一批日志交给 LLM 总结”缺少查询规划、反证、停止条件、引用和恢复能力，结果难以审计。

IncidentCopilot 将调查过程建模为受预算约束的 LangGraph 工作流：接收已经结构化的事故
范围与自然语言描述、规划、并行取证、生成与验证假设、判断证据充分性、必要时追加研究，
最后输出可追溯报告并支持人工复核。

## 3. 产品目标

### 3.1 必须达成

- **Current**：调用方提交自然语言故障描述，并显式提供单个 primary service 与带时区
  `start_time/end_time`；症状、严重性和环境是结构化可选字段。
- **Current 限制**：系统不从 raw query 自动推断 service 或 time window，也不实现新的
  自然语言 parser；缺少必填范围时 API 返回校验错误而不是猜测。
- 同轮并行查询六类证据：日志、指标、Trace、变更、拓扑、知识。
- 每条结论关联支持证据、反对证据和来源引用。
- 证据不足时产生下一轮定向查询，并受研究轮数、工具调用、Token 和时间预算约束。
- 部分数据源失败时保留错误并生成带局限说明的报告。
- 无外部服务、无 API Key 时使用 fixture、Fake Model 和 Fake Embedding 完整演示。
- 通过 Provider Protocol 替换为 Prometheus、Loki、Tempo 等真实适配器。
- 提供 checkpoint、thread ID、流式进度和人工接受/追加调查能力。
- 提供可复现的离线评估数据、指标计算和未美化的实际结果。

### 3.2 成功信号

项目完成时应能进行一段 10–15 分钟演示：

1. 提交 `payment-service` 错误率与超时问题。
2. 观察 Agent 并行调用多源工具并展示结构化事件。
3. 查看至少一个假设被反证或降权。
4. 查看带引用的根因报告、调查统计和不确定性。
5. 展示同一图在 fixture 模式下可复现，并说明真实 Provider 的替换点。
6. 运行测试与小型离线评估，明确区分测试通过与模型质量指标。

## 4. 用户与核心场景

### 4.1 主要角色

- 候选人/开发者：本地运行、调试、扩展 Provider、讲解设计取舍。
- 故障调查者：提交事件、观察进度、审阅证据与报告、要求追加调查。
- 面试官/评审者：检查架构边界、Graph 循环、测试、评估与可复现性。

### 4.2 基准演示事件

```text
payment-service 在 10:20 到 10:40 之间错误率突然升高，
部分支付请求超时，请分析根因并给出处理建议。
```

Fixture 的预期真相在数据集中显式标注，但不直接暴露给 Agent：近期发布导致数据库连接池上限错误，表现为连接获取超时；依赖服务的健康指标可用于排除外部支付网关为首因。最终数据内容在 Phase 2 定稿。

## 5. 功能需求

| ID | 需求 | 验收摘要 | 状态 / 历史目标 |
| --- | --- | --- | --- |
| FR-01 | 校验事故上下文 | 自然语言描述 + 调用方提供的单 primary service/带时区时间窗通过 Schema | Current |
| FR-02 | 制定调查计划 | 覆盖必要数据源并说明目的 | 4 |
| FR-03 | 并行多源取证 | 单个失败不取消其它分支 | 2、4 |
| FR-04 | 检索知识 | Hybrid Search、过滤、去重、引用 | 3 |
| FR-05 | 生成根因假设 | 一个或多个可验证假设 | 4 |
| FR-06 | 支持与反对证据 | 假设记录双向证据与验证查询 | 4 |
| FR-07 | 有界研究循环 | 达到充分性或任一预算上限即停止 | 4 |
| FR-08 | 结构化报告 | 覆盖需求列出的全部报告字段 | 4 |
| FR-09 | 调查 API | 创建、状态、报告接口 | 5 |
| FR-10 | 流式事件 | 发送节点、工具、预算、审核事件 | 5 |
| FR-11 | checkpoint/HITL | 可用 thread ID 恢复并接受/追加研究 | 5 |
| FR-12 | 离线 Evaluation | 数据集、Runner、逐项与聚合指标 | 6 |
| FR-13 | 真实数据源 | payment 场景的 Prometheus metrics Provider 可切换 | Current（窄场景） |

## 6. 报告契约

最终报告至少包含：故障摘要、最可能根因、置信度、影响服务、时间线、支持证据、反对证据、已排除假设、修复建议、风险、引用、调查过程摘要，以及 Token、工具调用次数、研究轮数和总耗时。

置信度表示“当前证据下的校准判断”，不是数学概率保证。报告必须同时给出：

- 置信度形成依据；
- 缺失或失败的数据源；
- 达到预算上限时尚未验证的问题；
- 建议人工确认的高风险操作。

## 7. 非功能需求

| 维度 | 要求 |
| --- | --- |
| 可运行性 | 默认 fixture 模式不需要网络、数据库或 API Key |
| 可复现性 | 固定 fixture、Fake Model、Fake Embedding；依赖由 `uv.lock` 精确锁定 |
| 性能 | Fixture 单次调查目标 P95 < 10 秒，不含人工等待；真实基线到 Phase 6 实测 |
| 可靠性 | Provider 超时、有限重试、错误归一化、部分结果可用 |
| 安全 | 工具参数白名单/边界、日志脱敏、秘密仅从环境注入、无自动修复操作 |
| 可观察 | 结构化日志、correlation/thread ID、节点与工具时延、预算使用、错误分类 |
| 可维护 | 清晰分层、Protocol 驱动、依赖注入、领域模型与持久化模型分离 |
| 可测试 | 默认无网络单测；图路径、故障降级、恢复和评估均有确定性测试 |
| 可访问性 | API 错误结构一致；演示 UI 若实现需有键盘可用与清晰状态文本 |

所有性能和质量数字必须由后续真实测试产生；上表中的 P95 是验收目标，不是当前结果。

## 8. 范围边界

### 8.1 MVP 范围内

- 单个事故、一个调用方提供的 primary service 的只读诊断。
- Fixture 全链路和至少一个真实可观测数据源适配器。
- 英文/中文自然语言描述与结构化调查范围；当前不从描述自动解析服务或时间窗。
- 小型知识库、离线评估集、API 与可选演示 UI。
- 人工审核报告及要求追加研究。

### 8.2 明确不做

- 自动执行扩容、回滚、重启、改配置等生产写操作。
- 生产级多租户、RBAC、计费、海量数据治理或高可用部署。
- 代替 SRE 作最终事故判定或承诺零误诊。
- 在 MVP 同时完成所有云厂商与可观测平台集成。
- 训练自有基础模型或在线微调。
- 把原始日志全文无限写入 Graph State 或发送给 LLM。

## 9. 风险与缓解

| 风险 | 缓解策略 |
| --- | --- |
| LLM 幻觉或忽略反证 | Schema 校验、引用完整性检查、确定性规则辅助、明确不确定性 |
| State 膨胀 | 原始证据外置；State 保存 ID、有界摘要、去重 reducer |
| 并行分支结果冲突 | 按证据 ID 去重的 reducer；同 superstep 只写声明过的聚合字段 |
| 循环失控/成本失控 | 四类硬预算与单节点超时；路由优先检查预算 |
| Fixture 过于理想化 | 包含噪声、矛盾证据和 Provider 故障场景；公开 ground truth |
| 依赖 API 快速变化 | 兼容范围 + 锁文件；Graph API 适配集中；升级需回归图测试 |
| RAG 指标看似漂亮 | 数据集版本化、逐样本输出、禁止只报告聚合结果 |

## 10. Current、Experimental 与 Target

| 状态 | 能力 |
| --- | --- |
| Current | Fixture/Fake 默认链路；单 primary service；FastAPI/SSE/HITL；内存任务事件仓储；Memory/PostgreSQL checkpoint；payment-only Prometheus metrics 演示 |
| Experimental | `PgVectorStore` 参数化 SQL Adapter；仅 recording fake 合同测试，不在默认 RAG 或 Compose 运行链路 |
| Target | 真实 LLM/Embedding、Loki/Tempo、完整多服务联合调查、持久 Investigation/Event/Evidence Store、生产鉴权与分布式执行 |

## 11. 当前产品级完成定义

- Current 能力的强制质量门禁通过，且路线图/进度保留真实记录。
- 新机器按 README 可启动 fixture 演示并运行测试。
- API、Graph、RAG、Provider 和 Evaluation 均至少有一条端到端验证路径。
- 报告中的引用可以解析回具体证据或知识块。
- 真实演示和离线评估数据可复现，没有虚构指标。
- 已知限制、失败模式和下一步扩展清晰记录。
